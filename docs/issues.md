# Анализ проблем проекта Meme Stealer

## Критические проблемы

### 1. Жёстко закодированные идентификаторы чатов
**Файлы:** `src/index.ts`, `src/modules/event-handler.ts`
**Проблема:** `PROXY_CHAT_ID` и `FILTERED_CHAT_ID` захардкожены в двух разных файлах. При изменении нужно править в нескольких местах.
**Решение:** Вынести в переменные окружения или единый конфигурационный файл.

### 2. Захардкоженная конфигурация прокси
**Файл:** `src/utils/config.ts`
**Проблема:** `SOCKS5_PROXY = { host: '127.0.0.1', port: 1081 }` захардкожен. Невозможно запустить без локального прокси или изменить адрес без правки кода.
**Решение:** Вынести `PROXY_HOST` и `PROXY_PORT` в `.env`, добавить возможность отключения прокси.

### 3. Отсутствие graceful shutdown
**Файл:** `src/index.ts`
**Проблема:** При ошибках вызывается `process.exit(1)`. Нет обработчиков сигналов `SIGTERM`/`SIGINT`. При остановке PM2 или контейнера:
- Не закрывается соединение с Telegram
- Не отключается Prisma клиент
- Возможна потеря данных или блокировка сессии
**Решение:** Добавить обработчики сигналов, корректно завершать polling, вызывать `db.$disconnect()` и `tgClient.disconnect()`.

### 4. Отсутствие валидации переменных окружения
**Файл:** `src/utils/config.ts`
**Проблема:** Если `API_ID`, `API_HASH` или другие критичные переменные не заданы, приложение упадёт с непонятной ошибкой в рантайме. `Number(undefined)` вернёт `NaN`.
**Решение:** Добавить валидацию через Zod или ручные проверки на старте с понятными сообщениями об ошибках.

### 5. Дублирование кода между polling и event-handler
**Файлы:** `src/index.ts`, `src/modules/event-handler.ts`
**Проблема:** Функции `hasUrl`, `extractThumbnailBytes`, `computeMd5`, логика пересылки дублируются в двух файлах. При изменении логики нужно править в двух местах.
**Решение:** Вынести общую логику в отдельные утилиты (`src/utils/message-utils.ts`), использовать в обоих модулях.

## Архитектурные проблемы

### 6. Последовательный опрос каналов (блокирующий)
**Файл:** `src/index.ts`, функция `pollMessages`
**Проблема:** Каналы опрашиваются последовательно в цикле `for...of`. Если каналов 20 и каждый запрос занимает 2 секунды, полный цикл = 40 секунд. При `POLL_INTERVAL = 60000` это терпимо, но при росте числа каналов задержки будут накапливаться.
**Решение:** Использовать `Promise.allSettled` для параллельного опроса с ограничением конкурентности (например, через p-limit или async-queue).

### 7. Отсутствие backoff при ошибках API
**Файл:** `src/index.ts`
**Проблема:** При ошибке запроса к Telegram API (rate limit, временная недоступность) polling просто продолжается через фиксированный интервал. Нет экспоненциальной задержки, что может усугубить проблему.
**Решение:** Реализовать exponential backoff с jitter при повторных ошибках.

### 8. Потенциальная гонка данных при обновлении lastMessageId
**Файл:** `src/index.ts`, строка `await updateLastMessageId(chatId, newestMessageId)`
**Проблема:** `newestMessageId` берётся из первого сообщения в ответе API, а не из последнего обработанного. Если сообщения приходят не по порядку или часть обработки упала с ошибкой, `lastMessageId` может перескочить вперёд, и непроцессированные сообщения будут пропущены навсегда.
**Решение:** Обновлять `lastMessageId` только после успешной обработки всех сообщений, использовать `Math.max(processedIds)`.

### 9. Динамические импорты внутри функций
**Файл:** `src/modules/event-handler.ts`
**Проблема:** `const { tgClient } = await import('@config')` вызывается внутри `forwardToFiltered` и `processMessage` при каждом сообщении. Это неэффективно и создаёт задержки.
**Решение:** Импортировать `tgClient` статически в начале файла, как это сделано в `src/index.ts`.

### 10. In-memory кэш админов без синхронизации
**Файл:** `src/services/admin.service.ts`
**Проблема:** `adminCache` хранится в памяти процесса. Если запустить несколько инстансов приложения (например, для отказоустойчивости), кэши рассинхронизируются. При добавлении/удалении админа изменения видны только в одном процессе.
**Решение:** Для single-instance это приемлемо, но стоит задокументировать ограничение. Для multi-instance использовать Redis или проверять БД при каждом запросе.

## Проблемы безопасности

### 11. Зависимости в неправильной категории
**Файл:** `package.json`
**Проблема:** Пакеты разработки находятся в `dependencies` вместо `devDependencies`:
- `eslint`, `eslint-config-prettier`, `eslint-plugin-import`, `eslint-plugin-prettier`
- `prettier`, `husky`, `nodemon`, `ts-jest`, `typescript`, `typescript-eslint`
Это увеличивает размер production-образа и время установки.
**Решение:** Переместить в `devDependencies`.

### 12. Отсутствие .env.example
**Проблема:** Нет шаблона с перечнем всех необходимых переменных окружения. Новому разработчику сложно понять, что нужно настроить.
**Решение:** Создать `.env.example` с комментариями и placeholder-значениями.

### 13. Сессия Telegram в переменной окружения
**Проблема:** `STRING_SESSION` даёт полный доступ к аккаунту Telegram. При утечке `.env` злоумышленник получает полный контроль.
**Решение:** Задокументировать риски, добавить предупреждение в README, рассмотреть шифрование сессии на диске.

### 14. Отсутствие rate limiting для бота
**Файл:** `src/bot/commands.ts`
**Проблема:** Нет защиты от спама командами. Злоумышленник может завалить бота запросами.
**Решение:** Добавить простой rate limiter (например, не более 5 команд в минуту на пользователя).

## Проблемы надёжности

### 15. Нет health check endpoint
**Проблема:** Невозможно externally проверить, жив ли бот и работает ли polling. PM2 считает процесс живым, даже если polling завис или Telegram отключился.
**Решение:** Поднять лёгкий HTTP-сервер (Express уже в зависимостях) с endpoint `/health`, который проверяет статус подключения и время последнего успешного polling.

### 16. Ошибки пересылки не обрабатываются
**Файл:** `src/index.ts`, функции `forwardToProxy`, `forwardToFiltered`
**Проблема:** Если `tgClient.forwardMessages` падает (канал удалён, бот заблокирован, нет прав), ошибка пробрасывается вверх и может прервать обработку остальных сообщений.
**Решение:** Обернуть forward в try-catch, логировать ошибку, продолжать обработку.

### 17. Потенциальная утечка памяти
**Файл:** `src/index.ts`, функция `pollMessages`
**Проблема:** `groups` Map создаётся каждый цикл, но если обработка зависает, старые Map могут накапливаться. Также `adminCache` никогда не очищается.
**Решение:** Убедиться, что все асинхронные операции завершаются, добавить таймауты на обработку.

### 18. Нет логирования в файл
**Файл:** `src/utils/system-logger.ts`
**Проблема:** Все логи выводятся только в консоль. При перезапуске процесса или ротации логов PM2 история теряется.
**Решение:** Добавить запись в файл (winston, pino, или простой fs.appendStream), настроить ротацию.

## Проблемы качества кода

### 19. Отсутствуют тесты
**Проблема:** В проекте нет ни одного теста. Невозможно безопасно рефакторить или добавлять функции без риска сломать существующую логику.
**Решение:** Начать с unit-тестов для чистых функций (`hasUrl`, `computeMd5`, `hammingDistance`), затем интеграционные тесты для БД.

### 20. TypeScript не в strict mode
**Файл:** `tsconfig.json`
**Проблема:** Отсутствуют `strict`, `strictNullChecks`, `noImplicitAny`. Это позволяет скрытым багам (null/undefined) проскальзывать в рантайм.
**Решение:** Включить `strict: true` и постепенно исправить возникающие ошибки типов.

### 21. Неправильный rootDir в tsconfig
**Файл:** `tsconfig.json`
**Проблема:** `"rootDir": "./"` включает все файлы проекта, включая `node_modules`, `docs`, `prisma`. Это может вызывать конфликты и замедлять компиляцию.
**Решение:** Установить `"rootDir": "./src"` или явно контролировать через `include`.

### 22. Дедупликация альбомов может пропускать контент
**Файл:** `src/index.ts`, функция `processGroupedMessages`
**Проблема:** Если первое сообщение в альбоме не имеет миниатюры (например, текстовое описание), `firstMd5` остаётся `null`, и вся группа пересылается без дедупликации, даже если остальные сообщения имеют миниатюры.
**Решение:** Искать первую миниатюру в группе до принятия решения, а не полагаться на порядок сообщений.

### 23. Отсутствие CI/CD
**Проблема:** Нет автоматической проверки линтером, типами или тестами при пуше. Ошибки обнаруживаются только в рантайме.
**Решение:** Добавить GitHub Actions workflow с `npm run build` (включает eslint + tsc).

## Известные ограничения (задокументированы, но требуют решения)

### 24. MD5 дедупликация не ловит визуально одинаковые мемы
**Файлы:** `docs/hash-problem.md`, `docs/perception-hash-plan.md`
**Статус:** Задокументировано, есть план перехода на pHash/dHash.
**Приоритет:** Высокий — основная функциональная ограниченность бота.

### 25. Видео обрабатываются только по миниатюре
**Проблема:** Для видео хешируется только thumbnail, а не содержимое. Два разных видео с одинаковой обложкой будут считаться дубликатами.
**Решение:** Для видео использовать отдельную логику (хеширование первого кадра или метаданных).
